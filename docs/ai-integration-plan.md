## 1. Executive Summary

**Good news first:** the foundations are already in place. `OPENAI_API_KEY` is stubbed in [.env.example:28](../.env.example#L28). The Pro-gating helper [getUserIsPro](../src/lib/billing.ts#L18) re-reads `isPro` from the DB on every write path (the same defense-in-depth pattern that protects `createItemAction` against stale JWTs). The rate-limit module at [src/lib/rate-limit.ts](../src/lib/rate-limit.ts) is a generic `LimiterName`-keyed Upstash wrapper that already fails open on outages — adding AI limiters is a one-line registry edit per feature. The `Item` model already carries `content`, `description`, `language`, and `itemType.name`, so every AI feature has the inputs it needs without a schema change.

**What's missing:** the `openai` SDK is not installed (no entry in `package.json`). No `src/lib/ai.ts`, no `src/lib/ai-prompts.ts`, no AI server actions or routes. The item drawer's action bar has no AI affordances yet — none of [item-drawer-action-bars.tsx](../src/components/items/item-drawer-action-bars.tsx) mentions OpenAI. The "Try AI auto-tagging on an item" checklist row in `/checkout/success` points at an `/ai-tagging` link that 404s by design (see the Welcome to Pro feature entry).

**Approach:** OpenAI Node SDK direct (not Vercel AI SDK — see [§4](#4-sdk-choice)) on `gpt-5-nano` as the default model. Use **Server Actions** for the structured-output features (auto-tag, summary) and **Route Handlers** for the streaming features (explain, optimize). Gate everything behind `getUserIsPro` and a per-user daily Upstash limiter. Validate every AI output with Zod via the SDK's `zodResponseFormat` helper so callers never have to defensively parse JSON.

## 3. Model & Cost Strategy

### 3.1 Recommended model: `gpt-5-nano`

| Property | Value | Why it matters |
| --- | --- | --- |
| Input price | **$0.050 / 1M tokens** | ~3× cheaper than `gpt-4o-mini` ($0.15/M) for the same call shape |
| Output price | **$0.400 / 1M tokens** | ~1.5× cheaper than `gpt-4o-mini` ($0.60/M) |
| Context window | 400K tokens | We will never get close — items are typically <2K tokens |
| Structured outputs | ✓ (strict-mode) | Required for auto-tagging — Zod schema → JSON schema, no defensive parsing |
| Streaming | ✓ | Required for Explain and Optimize |
| Vision / tools | ✓ | Not used in v1; available if we add image-snippet OCR later |

Sources: [GPT-5 nano Model · OpenAI API docs](https://developers.openai.com/api/docs/models/gpt-5-nano), [OpenAI API Pricing](https://openai.com/api/pricing/).

### 3.2 Cost back-of-envelope

Assume a busy Pro user runs ~30 AI actions/day:

- 20× auto-tag (≈400 in / 60 out tokens) → 8K in / 1.2K out = $0.0004 + $0.00048 = **$0.00088**
- 5× summary (≈600 in / 40 out) → 3K in / 0.2K out = $0.00015 + $0.00008 = **$0.00023**
- 3× explain code (≈800 in / 400 out) → 2.4K in / 1.2K out = $0.00012 + $0.00048 = **$0.00060**
- 2× optimize prompt (≈500 in / 500 out) → 1K in / 1K out = $0.00005 + $0.0004 = **$0.00045**

**Per heavy user / day: ~$0.002. Per heavy user / month: ~$0.06.** Pro is $8/mo, so unit economics are fine. The risk is not aggregate cost — it's abuse spikes from a single user. That risk is mitigated by [§8 rate limits](#8-pro-gating--rate-limiting) and [§12 input clamps](#12-cost-optimization).

### 3.3 Spec deviation — `gpt-4o-mini` → `gpt-5-nano`

The [project overview](../context/project-overview.md#L330) and [AI Features section](../context/project-overview.md#L335-L344) name `gpt-4o-mini`. `gpt-5-nano` (released Aug 2025) supersedes it at lower cost with the same capabilities for our use cases. Recommendation: bump the spec to `gpt-5-nano` in the same PR that ships this plan's Phase 1.

If pricing or quality changes, the model name lives in **one place** ([§7.1](#71-srclibconstantsts-new-constants)) so a swap is a single-line edit.

## 4. SDK Choice

**Recommendation: OpenAI Node SDK direct (no Vercel AI SDK wrapper).**

| Question | Answer |
| --- | --- |
| Are we multi-provider? | No. The roadmap is OpenAI-only. |
| Do we need React/Next streaming primitives? | Modest — only 2 of 4 features stream; both are short outputs that don't need `useChat`/`useCompletion` ergonomics. |
| Do we want a server-side abstraction over `Stripe`/`Upstash`-style SDK setup? | Yes — and we already have that pattern (`getStripe()`). Adding the AI SDK on top duplicates it. |
| Bundle/edge concerns? | AI server actions run in Node runtime (Prisma + Upstash already do). No edge constraint. |
| Vercel AI SDK upsides we'd lose? | Built-in `useChat`/`useCompletion` React hooks; `generateObject`/`streamObject` higher-level helpers; provider-swap. None are load-bearing for v1. |

The OpenAI SDK at v5+ already ships `zodResponseFormat()`, `zodTextFormat()`, `zodFunction()` helpers that hand us typed structured outputs end-to-end ([helpers.md in openai-node](https://github.com/openai/openai-node/blob/master/helpers.md)) — the main thing the AI SDK was giving us a year ago. We get the type safety with one fewer dependency in the lockfile.

If we ever add a second provider (Anthropic, local Ollama) the AI SDK becomes the right answer. Until then, direct is cleaner.

## 6. Files to Create

### 6.1 `src/lib/ai.ts`

Lazy singleton. Mirrors [src/lib/stripe.ts](../src/lib/stripe.ts) exactly.

```typescript
import "server-only";

import OpenAI from "openai";

import { AI_MODEL } from "@/lib/constants";

let cached: { client: OpenAI; apiKey: string } | null = null;

function requireEnv(name: string): string {
  const value = process.env[name];
  if (!value) throw new Error(`${name} is not configured.`);
  return value;
}

export function getOpenAI(): OpenAI {
  const apiKey = requireEnv("OPENAI_API_KEY");
  if (cached && cached.apiKey === apiKey) return cached.client;
  // `maxRetries: 2` is the SDK default; setting explicitly to document intent.
  const client = new OpenAI({ apiKey, maxRetries: 2, timeout: 60_000 });
  cached = { client, apiKey };
  return client;
}

export { AI_MODEL };
```

import "server-only";

import type { SystemTypeName } from "@/lib/system-types";

export const SYSTEM_PROMPTS = {
  tags: `You are a tagging assistant for a developer knowledge hub.
Given the user's content, suggest 3-7 lowercase, kebab-case tags.
Tags MUST be short (1-3 words), specific, and useful for filtering.
Use existing-tag conventions when relevant: react, typescript, postgres, prompt-engineering.
NEVER include sensitive data (emails, tokens, secrets) in tags.

  summary: `You are a summarizer for a developer knowledge hub.

  explain: `You are a code-explanation assistant for a developer.
Explain in plain English what the code does, how it works, and any non-obvious behavior.
Be concise — 2-4 short paragraphs maximum. Use markdown headings and code spans where helpful.
Do NOT rewrite the code. Do NOT add usage examples unless asked.`,

  optimize: `You are a prompt-optimization assistant.
Rewrite the user's AI prompt to be clearer, more specific, and more likely to produce the desired output.
Preserve the user's intent and any concrete constraints they specified.
Return ONLY the rewritten prompt — no preamble, no explanation, no markdown.`

// Per-type prompt selector for summaries (a snippet asks "what does this do?",
// a prompt asks "what is this prompt for?", etc.). Tags/explain/optimize don't
// need per-type variants in v1.
export const SUMMARY_BY_TYPE: Partial<Record<SystemTypeName, string>> = {
  snippet: "Summarize what this code does.",
  prompt: "Summarize what this AI prompt is for.",
  command: "Summarize what this terminal command does.",

### 6.3 `src/lib/ai-usage.ts`

Per-user daily AI quota — keyed by `userId`, separate from the per-feature spike-protection limiters in [§8](#8-pro-gating--rate-limiting). Mirrors [`checkItemCapacity`](../src/lib/billing.ts#L29-L42) shape so callers handle one discriminated union.

```typescript
import "server-only";

import { rateLimit } from "@/lib/rate-limit";

export type AiQuotaCheck = { ok: true } | { ok: false; reason: string };

export async function checkAiQuota(userId: string): Promise<AiQuotaCheck> {
  // "aiDaily" is registered in src/lib/rate-limit.ts LIMITS — 100/day per user.
  const result = await rateLimit("aiDaily", `user:${userId}`);
  if (result.success) return { ok: true };
  return {
    ok: false,
    reason: "You've hit today's AI usage limit. Try again tomorrow."
  };
}
```

### 6.4 `src/lib/ai-truncate.ts`

Hard input clamps so a 50K-line snippet can't generate a $5 API call.

```typescript
import "server-only";

import { AI_MAX_INPUT_CHARS } from "@/lib/constants";

export function clampForAi(value: string): string {
  if (value.length <= AI_MAX_INPUT_CHARS) return value;
  return value.slice(0, AI_MAX_INPUT_CHARS) + "\n…[truncated]";
}
```

### 6.5 `src/actions/ai/tags.ts`

Server action — non-streaming, returns `string[]`. Drawer shows suggestions as chips; clicking a chip adds the tag to `EditState.tags`. Drawer's existing Save flow writes via `updateItemAction` — no new write path.

```typescript
"use server";

import { z } from "zod";
import { zodResponseFormat } from "openai/helpers/zod";

import { auth } from "@/auth";
import { getOpenAI } from "@/lib/ai";
import { SYSTEM_PROMPTS } from "@/lib/ai-prompts";
import { clampForAi } from "@/lib/ai-truncate";
import { checkAiQuota } from "@/lib/ai-usage";
import { getUserIsPro } from "@/lib/billing";
import { rateLimit, rateLimitMessage } from "@/lib/rate-limit";
import { AI_MODEL } from "@/lib/constants";

const TagsSchema = z.object({
  tags: z
    .array(z.string().min(1).max(40).regex(/^[a-z0-9-]+$/))
    .min(1)
    .max(7)
});

const inputSchema = z.object({
  title: z.string().max(200),
  content: z.string().nullable(),
  description: z.string().nullable(),
  language: z.string().nullable(),
  typeName: z.string()
});

export type SuggestTagsResult =
  | { success: true; tags: string[] }
  | { success: false; error: string };

export async function suggestTagsAction(
  payload: z.input<typeof inputSchema>
): Promise<SuggestTagsResult> {
  const session = await auth();
  if (!session?.user?.id) return { success: false, error: "You are not signed in." };

  const isPro = await getUserIsPro(session.user.id);
  if (!isPro) return { success: false, error: "AI features require Pro." };

  const perFeature = await rateLimit("aiTags", `user:${session.user.id}`);
  if (!perFeature.success) {
    return { success: false, error: rateLimitMessage(perFeature) };
  }
  const daily = await checkAiQuota(session.user.id);
  if (!daily.ok) return { success: false, error: daily.reason };

  const parsed = inputSchema.safeParse(payload);
  if (!parsed.success) return { success: false, error: "Invalid input." };

  const userBlock = [
    `Type: ${parsed.data.typeName}`,
    parsed.data.language ? `Language: ${parsed.data.language}` : null,
    `Title: ${parsed.data.title}`,
    parsed.data.description ? `Description: ${parsed.data.description}` : null,
    parsed.data.content
      ? `Content:\n${clampForAi(parsed.data.content)}`
      : null
  ]
    .filter(Boolean)
    .join("\n");

  try {
    const completion = await getOpenAI().chat.completions.parse({
      model: AI_MODEL,
      messages: [
        { role: "system", content: SYSTEM_PROMPTS.tags },
        { role: "user", content: userBlock }
      ],
      response_format: zodResponseFormat(TagsSchema, "tags"),
      temperature: 0.4
    });
    const parsedTags = completion.choices[0]?.message.parsed;
    if (!parsedTags) return { success: false, error: "AI did not return tags." };
    return { success: true, tags: Array.from(new Set(parsedTags.tags)) };
  } catch (error) {
    console.error("suggestTagsAction failed", error);
    return { success: false, error: "Could not suggest tags. Please try again." };
  }
}
```

### 6.7 `src/app/api/ai/explain/route.ts` (streaming)

Streaming → can't use Server Action ergonomically; use a Route Handler that returns a Web `ReadableStream` of plain text. Both Explain and Optimize stream because output is multi-paragraph and feels slow without it.

```typescript
import { NextRequest } from "next/server";
import { z } from "zod";

import { auth } from "@/auth";
import { getOpenAI } from "@/lib/ai";
import { SYSTEM_PROMPTS } from "@/lib/ai-prompts";
import { clampForAi } from "@/lib/ai-truncate";
import { checkAiQuota } from "@/lib/ai-usage";
import { getUserIsPro } from "@/lib/billing";
import { rateLimit, rateLimitJsonResponse } from "@/lib/rate-limit";
import { AI_MODEL } from "@/lib/constants";

export const runtime = "nodejs";

const bodySchema = z.object({
  code: z.string().min(1).max(50_000),
  language: z.string().optional().nullable()
});

export async function POST(request: NextRequest) {
  const session = await auth();
  if (!session?.user?.id) {
    return new Response(JSON.stringify({ error: "Unauthorized" }), { status: 401 });
  }

  const isPro = await getUserIsPro(session.user.id);
  if (!isPro) {
    return new Response(
      JSON.stringify({ error: "AI features require Pro." }),
      { status: 403 }
    );
  }

  const perFeature = await rateLimit("aiExplain", `user:${session.user.id}`);
  if (!perFeature.success) return rateLimitJsonResponse(perFeature);
  const daily = await checkAiQuota(session.user.id);
  if (!daily.ok) {
    return new Response(JSON.stringify({ error: daily.reason }), { status: 429 });
  }

  let body: z.infer<typeof bodySchema>;
  try {
    body = bodySchema.parse(await request.json());
  } catch {
    return new Response(JSON.stringify({ error: "Invalid input." }), { status: 400 });
  }

  const stream = await getOpenAI().chat.completions.create({
    model: AI_MODEL,
    stream: true,
    messages: [
      { role: "system", content: SYSTEM_PROMPTS.explain },
      {
        role: "user",
        content: `Language: ${body.language ?? "unknown"}\n\n${clampForAi(body.code)}`
      }
    ],
    temperature: 0.4
  });

  const encoder = new TextEncoder();
  const readable = new ReadableStream({
    async start(controller) {
      try {
        for await (const chunk of stream) {
          const delta = chunk.choices[0]?.delta?.content;
          if (delta) controller.enqueue(encoder.encode(delta));
        }
        controller.close();
      } catch (err) {
        console.error("[ai/explain] stream failed", err);
        controller.error(err);
      }
    }
  });

  return new Response(readable, {
    headers: {
      "Content-Type": "text/plain; charset=utf-8",
      "Cache-Control": "no-store",
      "X-Content-Type-Options": "nosniff"
    }
  });
}
```

## 8. Pro Gating + Rate Limiting

Defense layers, top to bottom:

1. **UI gate (display only)** — `session.user.isPro` decides whether AI buttons render. Cheap, instant, but trust-no-one. JWT can be stale across a Stripe downgrade for one request.
2. **Server-side Pro re-check (authoritative)** — every AI action/route calls `await getUserIsPro(session.user.id)` — DB read, never stale. Mirrors what [createItemAction:210](../src/actions/items.ts#L210) already does.
3. **Per-feature spike limiter** — Upstash sliding window (e.g. 30 tag suggestions/hour). Stops accidental loops in client code from running up a bill.
4. **Daily aggregate limiter** — `aiDaily` 100/day per user. Caps the worst-case daily spend per Pro user at ~$0.20 (well under their $8/mo ARPU even at 100% AI utilization).
5. **Input clamp** — `clampForAi()` hard-truncates payloads to `AI_MAX_INPUT_CHARS` so a one-shot 50K snippet can't dwarf the daily cap.
6. **Output cap** — `max_tokens` on every non-streaming call; streaming calls implicitly capped by the model's response patterns + temperature.

All six together mean: a logged-out user gets 401, a free user gets 403, a Pro user with a runaway client gets 429 (with `Retry-After`), and a Pro user with a runaway *prompt* gets a truncated input.

Key keying decision: limiters are keyed by **userId**, not IP. Pro users are authenticated by definition; IP keying would penalize office NAT and reward proxy abuse.

## 9. Streaming vs Non-Streaming

Pick per feature:

| Feature | Streaming? | Reason |
| --- | :-: | --- |
| Auto-tag | No | Output is 3-7 tags as JSON. Streaming partial JSON is awkward; structured output via `zodResponseFormat` is one round-trip and type-safe end-to-end. |
| Summary | No | One short sentence. Network + token latency are similar; streaming would add complexity for no UX win. |
| Explain Code | **Yes** | Multi-paragraph markdown. Time-to-first-token >> total time for long explanations; users perceive it as slow without streaming. |
| Optimize Prompt | **Yes** | Same as Explain — output is prose-length, users want to see it appear. |

The cost of "streaming" in our stack is one Route Handler instead of one Server Action. Both still use the same `getOpenAI()` singleton + same rate-limit module. Don't try to stream from Server Actions — Next 16 supports it but ergonomics are worse than a Route Handler with `fetch()` + `ReadableStream` on the client.

Client-side streaming reader pattern (for the dialog components):

```typescript
const res = await fetch("/api/ai/explain", { method: "POST", body: JSON.stringify(payload) });
if (!res.ok) { toast.error((await res.json()).error); return; }
const reader = res.body!.getReader();
const decoder = new TextDecoder();
let acc = "";
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  acc += decoder.decode(value, { stream: true });
  setText(acc); // re-render with progressive text
}
```

`AbortController` should wrap this fetch + abort on dialog close so a backgrounded request doesn't hold open an OpenAI connection.

## 10. Security & Prompt Injection Defense

User content in DevStash is data — never instructions. The model gets that ordering via:

1. **Role discipline.** System prompt = our instructions. User role = wrapped data with a clear preamble (`Title: …\nContent:\n…`). Never concatenate user content into the system prompt.
2. **Output validation.** `zodResponseFormat` enforces the tag shape (`^[a-z0-9-]+$`, 1-7 entries, ≤40 chars each). An adversarial prompt can't get `<script>alert(1)</script>` into a tag — the strict-mode parse rejects it.
3. **No tools, no agency.** None of the four features give the model `tools` access. It cannot mutate items, read other users' data, or fetch URLs. The blast radius of a successful injection is "the AI's response is weird in this one drawer", not data exfiltration.
4. **HTML/markdown sanitation on render.** Explain output renders through the existing `MarkdownEditor` with `react-markdown` + `remark-gfm` — no `rehype-raw`, no inline HTML. Optimize output renders as plain text.
5. **No PII in tags.** System prompt explicitly forbids emails/tokens in tags. The Zod regex makes this hard anyway (no `@`).
6. **Log discipline.** Server-side `console.error` lines should not include raw user content — only error class + sanitized message. Already the pattern in [items.ts:260](../src/actions/items.ts#L260) and friends.
7. **API key handling.** `OPENAI_API_KEY` is server-only (`"server-only"` import on `src/lib/ai.ts` enforces it at build time — any client component that transitively imports it fails the build).
8. **No client-direct calls to OpenAI.** All AI traffic goes through our routes/actions so we can rate-limit, log, and gate centrally. Never ship `dangerouslyAllowBrowser: true`.

Sources: [OpenAI agent-builder safety guide](https://platform.openai.com/docs/guides/agent-builder-safety), [OpenAI on prompt injection persistence (VentureBeat)](https://venturebeat.com/security/openai-admits-that-prompt-injection-is-here-to-stay/). The TL;DR from OpenAI's own writing: prompt injection is unsolvable in the general case. The mitigation is to ensure the LLM has no power to do harm even when it's wrong — that's why "no tools" is the most important line above.

## 11. UI Patterns

### 11.1 Loading

- **Tags / Summary** (non-streaming): button shows `LoaderCircle` spinner (same primitive used in [`PendingButton`](../src/components/ui/pending-button.tsx)). Use `useTransition` so the rest of the drawer stays interactive.
- **Explain / Optimize** (streaming): dialog opens immediately with skeleton text; first token replaces skeleton; cursor `▍` blinks at end of streamed text until `done`.

### 11.2 Accept / Reject

Tags as chips with `+` icon. Click = add. Already-present tag is shown muted with `Check` and not clickable. There's no "reject" — unwanted suggestions are simply not clicked. After 30s of no interaction, suggestions fade and a "Suggest again?" link replaces them.

Summary as a callout below the description with `Accept` (writes to `EditState.description` overwriting current value — confirm if non-empty) and `Dismiss`.

Optimize as a dialog showing **the rewritten prompt only** (per the spec — system prompt forbids preamble). `Use this version` button copies into `EditState.content`. `Discard` closes.

Explain as a dialog with `Copy` button and `Close`. No write back — explanations are reference, not content.

### 11.3 Error states

- 401 → toast "Sign in to use AI features." + close dialog.
- 403 → toast with `Upgrade` action button (same pattern as [NewItemDialog Pro-error toast](../src/components/items/new-item-dialog.tsx#L156-L170)) — routes to `/upgrade`.
- 429 → toast "You've hit your AI limit. Try again in {Retry-After}."
- Network/abort → silent on user-cancel; toast "AI request failed" on real failure.

### 11.4 Empty states

If the model returns 0 tags (rare with `min(1)` Zod constraint) or an empty summary: show "No suggestions for this item." in the chip area; don't break the action bar.

## 12. Cost Optimization

Beyond the input/output caps already in [§8](#8-pro-gating--rate-limiting):

1. **No multi-turn conversations.** Every feature is a one-shot. We never build conversation history into the call, never re-send prior messages. The system prompt is the only persistent context per request.
2. **`max_tokens` everywhere on non-streaming.** Tags ≤ 200 tokens, summary ≤ 80 tokens. Streaming responses don't get `max_tokens` (defeats the perceived-speed point) but Explain/Optimize are naturally bounded by the system prompt's "2-4 short paragraphs" / "no preamble" instructions.
3. **Temperature ~0.3-0.4.** Lower temperature = more focused = often fewer output tokens for the same task quality.
4. **No retries beyond the SDK default** (`maxRetries: 2`). A failed call is a toast, not a $0.50 retry storm.
5. **Cache nothing in v1.** Caching summaries against content hash is tempting but ~$0.0002 per call doesn't justify the cache-invalidation complexity. Revisit if monthly OpenAI bill exceeds $20.
6. **Truncate before clamp.** `clampForAi()` is the safety net; the prompt builder for tags already pre-truncates code blocks at line boundaries when possible (cleaner truncation than mid-token).
7. **Monitor.** Add a single `usage` log line per successful AI call: `[ai] tags userId=… in=… out=…` so we can grep monthly and project costs. Defer fancy observability (LangSmith / Helicone) until needed.

## 15. Open Questions / Deferred

- **Caching identical AI calls.** Hash `(systemPrompt, userPayload, model)` → store result in Redis with a 7-day TTL. Saves money on users who re-click the same suggestion. Defer until monthly OpenAI spend > $20/user.
- **AI-suggested collections.** "Add this item to the collection it probably belongs in." Not in the spec, but easy on top of summary. Defer to v1.1.
- **Bulk operations.** "Suggest tags for all my untagged items." Would need a job queue (none currently in the codebase). Defer.
- **Per-Pro-user tier of AI usage.** Currently flat 100/day across all Pro users. If we add a Pro+ tier, lift `AI_DAILY_REQUESTS_PER_USER` per-user via a `User.aiDailyLimitOverride` column. Schema change — defer.
- **Provider abstraction.** Vercel AI SDK or LangChain wrapper if we ever go multi-provider. Adds dependency surface area; not justified by v1 scope.
- **Embeddings / semantic search.** Currently search is fuzzy string match in-memory. Embeddings on items + pgvector would be a separate research doc.
- **Observability / cost dashboards.** Self-rolled `[ai] usage` log lines (see [§12](#12-cost-optimization)) are good enough for v1. Helicone / Langfuse / LangSmith if we need per-call inspection later.
- **Image OCR.** `gpt-5-nano` supports vision. We could auto-tag image-type items by alt-text generation. Defer to v1.1.
- **AI kill-switch env.** Single `AI_ENABLED=false` to short-circuit all four features without a code deploy if OpenAI has an outage we can't ride out. Trivial to add; defer until first incident.
