## 1. Executive Summary

**Good news first:** the `User` model already has `isPro`, `stripeCustomerId`, and `stripeSubscriptionId` columns (declared in [prisma/schema.prisma:23-25](../prisma/schema.prisma#L23-L25) and present in migration [20260427202202_init](../prisma/migrations/20260427202202_init/migration.sql)). The `.env.example` even has Stripe env-var stubs (with one typo to fix). **No schema migration is required.**

**What's missing:**

- `stripe` SDK is not installed (`package.json` has no `stripe` dependency).
- No checkout, portal, or webhook routes exist.
- `isPro` is not threaded through the NextAuth JWT/session — every code path currently treats users as full Pro (per [context/project-overview.md:466](../context/project-overview.md#L466)).
- Free-tier limits (50 items / 3 collections) are spec only — `createItemAction` and `createCollectionAction` have no count checks.
- The only Pro UI today is a hard-coded `"PRO"` badge on `file`/`image` sidebar entries ([src/components/dashboard/sidebar.tsx:93](../src/components/dashboard/sidebar.tsx#L93)) — purely cosmetic.
- The Settings page has no Billing section ([src/app/(dashboard)/settings/page.tsx](../src/app/(dashboard)/settings/page.tsx) — three cards: Account, Editor preferences, Danger zone).
- `deleteAccountAction` deletes the user row but **does not cancel the Stripe subscription** ([src/actions/account.ts:8-19](../src/actions/account.ts#L8-L19)).

**Approach:** Use **Stripe Checkout + Customer Portal** (hosted, low-surface-area). Avoid custom payment UI. One webhook endpoint reconciles `isPro` / `stripeSubscriptionId` from `customer.subscription.*` events.

### 2.6 API & server-action conventions

- **API routes** live at [src/app/api/.../route.ts](../src/app/api/), explicitly opt into Node runtime via `export const runtime = "nodejs";` when they touch Prisma/bcrypt/etc., parse JSON defensively (`try { body = await request.json() } catch {}`), return `NextResponse.json({ error }, { status })` on failure.
- **Rate limiting** is per-route via [src/lib/rate-limit.ts](../src/lib/rate-limit.ts) — `rateLimit("name", extractIp(request.headers))` → `rateLimitJsonResponse(limit)` on miss. The limiter is **per-name keyed** (cached `LIMITS` table), fails open on missing Upstash env. Add a new `LimiterName` entry per new public endpoint.
- **Server actions** live at [src/actions/](../src/actions/), all `"use server"` at top, returning `{ success: true, ... } | { success: false, error: string }` discriminated unions.
- **Zod 4** for all input validation.
- **Tests**: Vitest, server-actions + lib only (no components), Prisma + `@/auth` mocked at the module boundary. See [coding-standards.md](../context/coding-standards.md#testing).

## 3. Stripe Dashboard Setup

Manual steps (do once per environment — Test + Live):

1. **Create Product**: "DevStash Pro" in Stripe Dashboard → Products.
2. **Add Two Prices** to that product:
   - **Monthly**: $8.00 USD, recurring monthly. Save the `price_…` ID.
   - **Yearly**: $72.00 USD, recurring yearly. Save the `price_…` ID.
3. **Activate the Customer Portal**: Settings → Billing → Customer portal:
   - Enable cancellation (immediate or at period end — recommend *period end* so users keep Pro until the paid period expires).
   - Enable plan switching between monthly ↔ yearly.
   - Enable update payment method.
   - Set "Default redirect URL" to `https://<your-domain>/settings` (the portal sends users back here when they click "Return to DevStash").
4. **Enable Tax** (optional, recommended for production): Settings → Tax → Tax collection. If enabled, set `automatic_tax: { enabled: true }` on the Checkout Session.
5. **Configure Webhook Endpoint**: Developers → Webhooks → Add endpoint:
   - URL: `https://<your-domain>/api/stripe/webhook`
   - Events to subscribe to (minimum):
     - `checkout.session.completed`
     - `customer.subscription.created`
     - `customer.subscription.updated`
     - `customer.subscription.deleted`
     - `invoice.payment_failed` (optional — for surfacing dunning state)
   - Copy the **signing secret** (`whsec_…`) into `STRIPE_WEBHOOK_SECRET`.

> **Local dev:** Use the Stripe CLI to forward webhooks: `stripe listen --forward-to localhost:3000/api/stripe/webhook` — the CLI prints a separate `whsec_…` secret for local use. Put that in `.env`, not the production secret.

### 5.1 `src/lib/stripe.ts` — Stripe SDK singleton

```ts
import "server-only";
import Stripe from "stripe";

let cached: Stripe | null = null;

export function getStripe(): Stripe {
  if (cached) return cached;
  const key = process.env.STRIPE_SECRET_KEY;
  if (!key) throw new Error("STRIPE_SECRET_KEY is not set");
  cached = new Stripe(key, {
    apiVersion: "2025-09-30.acacia", // Pin to latest stable as of integration date — bump deliberately.
    typescript: true,
    appInfo: { name: "DevStash", version: "0.1.0" }
  });
  return cached;
}

export const STRIPE_PRICE_IDS = {
  monthly: () => requireEnv("STRIPE_PRICE_ID_MONTHLY"),
  yearly: () => requireEnv("STRIPE_PRICE_ID_YEARLY")
} as const;

export type BillingPeriod = keyof typeof STRIPE_PRICE_IDS;

function requireEnv(name: string): string {
  const value = process.env[name];
  if (!value) throw new Error(`${name} is not set`);
  return value;
}
```

Pattern mirrors [src/lib/r2.ts](../src/lib/r2.ts) (lazy singleton, env-thrown on use, not on import).

### 5.2 `src/lib/billing.ts` — Free-tier limit constants + helpers

```ts
import "server-only";
import { getItemStatsForUser } from "@/lib/db/items";
import { getCollectionStatsForUser } from "@/lib/db/collections";

export const FREE_TIER_LIMITS = {
  items: 50,
  collections: 3
} as const;

export const PRO_ONLY_ITEM_TYPES = new Set<string>(["file", "image"]);

export async function getUserIsPro(userId: string): Promise<boolean> {
  const { prisma } = await import("@/lib/prisma");
  const row = await prisma.user.findUnique({
    where: { id: userId },
    select: { isPro: true }
  });
  return row?.isPro ?? false;
}

export type CapacityCheck =
  | { ok: true }
  | { ok: false; reason: string };

export async function checkItemCapacity(userId: string, isPro: boolean): Promise<CapacityCheck> {
  if (isPro) return { ok: true };
  const { total } = await getItemStatsForUser(userId);
  if (total >= FREE_TIER_LIMITS.items) {
    return {
      ok: false,
      reason: `Free plan is limited to ${FREE_TIER_LIMITS.items} items. Upgrade to Pro for unlimited.`
    };
  }
  return { ok: true };
}

export async function checkCollectionCapacity(userId: string, isPro: boolean): Promise<CapacityCheck> {
  if (isPro) return { ok: true };
  const { total } = await getCollectionStatsForUser(userId);
  if (total >= FREE_TIER_LIMITS.collections) {
    return {
      ok: false,
      reason: `Free plan is limited to ${FREE_TIER_LIMITS.collections} collections. Upgrade to Pro for unlimited.`
    };
  }
  return { ok: true };
}
```

> Note: `getUserIsPro` uses a fresh DB read instead of trusting `session.user.isPro` — for write paths this avoids stale-session bypass after a downgrade (the JWT could be holding `isPro: true` from a webhook-stale token). For read/UI paths, the JWT-cached `isPro` is fine (see §8).

### 5.3 `src/actions/billing.ts` — Server actions for Checkout + Portal

```ts
"use server";
import { redirect } from "next/navigation";
import { auth } from "@/auth";
import { prisma } from "@/lib/prisma";
import { getStripe, STRIPE_PRICE_IDS, type BillingPeriod } from "@/lib/stripe";
import { getBaseUrl } from "@/lib/base-url";

export async function createCheckoutSessionAction(period: BillingPeriod): Promise<{ error: string } | void> {
  const session = await auth();
  if (!session?.user?.id || !session.user.email) {
    return { error: "You must be signed in." };
  }
  if (period !== "monthly" && period !== "yearly") {
    return { error: "Invalid billing period." };
  }

  const user = await prisma.user.findUnique({
    where: { id: session.user.id },
    select: { stripeCustomerId: true, email: true, isPro: true }
  });
  if (!user) return { error: "User not found." };
  if (user.isPro) return { error: "You already have an active subscription." };

  const stripe = getStripe();
  const baseUrl = await getBaseUrl();

  // Reuse the existing customer if we have one; otherwise let Checkout create one.
  // We DON'T pre-create a customer here — Checkout creating it (and us reading it
  // back via the webhook) is simpler and avoids dangling unused customers.
  const checkout = await stripe.checkout.sessions.create({
    mode: "subscription",
    customer: user.stripeCustomerId ?? undefined,
    customer_email: user.stripeCustomerId ? undefined : user.email,
    client_reference_id: session.user.id, // Resolves user from webhook even if customer is brand-new.
    line_items: [{ price: STRIPE_PRICE_IDS[period](), quantity: 1 }],
    allow_promotion_codes: true,
    billing_address_collection: "auto",
    automatic_tax: { enabled: false }, // Flip to true once Stripe Tax is configured.
    success_url: `${baseUrl}/settings?checkout=success`,
    cancel_url: `${baseUrl}/settings?checkout=cancelled`,
    subscription_data: { metadata: { userId: session.user.id } }
  });

  if (!checkout.url) {
    return { error: "Could not start checkout. Please try again." };
  }
  redirect(checkout.url);
}

export async function createPortalSessionAction(): Promise<{ error: string } | void> {
  const session = await auth();
  if (!session?.user?.id) return { error: "You must be signed in." };

  const user = await prisma.user.findUnique({
    where: { id: session.user.id },
    select: { stripeCustomerId: true }
  });
  if (!user?.stripeCustomerId) {
    return { error: "No active subscription found. Subscribe to Pro first." };
  }

  const stripe = getStripe();
  const baseUrl = await getBaseUrl();
  const portal = await stripe.billingPortal.sessions.create({
    customer: user.stripeCustomerId,
    return_url: `${baseUrl}/settings`
  });
  redirect(portal.url);
}
```

> **Why `redirect()` not `return { url }`?** `redirect()` from server actions is the documented Next 16 pattern — it produces a 303 to the client without exposing the Stripe URL in the response body. Avoids the "form submit → wait for JSON → window.location" round-trip and stays compatible with progressive-enhancement form posts.

### 5.4 `src/app/api/stripe/webhook/route.ts` — Webhook handler

```ts
import { NextResponse } from "next/server";
import type Stripe from "stripe";
import { headers } from "next/headers";

import { getStripe } from "@/lib/stripe";
import { prisma } from "@/lib/prisma";

export const runtime = "nodejs"; // Stripe SDK needs Node.

export async function POST(request: Request) {
  const secret = process.env.STRIPE_WEBHOOK_SECRET;
  if (!secret) {
    console.error("[stripe/webhook] STRIPE_WEBHOOK_SECRET is not set");
    return NextResponse.json({ error: "Webhook not configured" }, { status: 500 });
  }

  // Stripe needs the raw body to verify the signature; do NOT call .json() first.
  const body = await request.text();
  const sig = (await headers()).get("stripe-signature");
  if (!sig) {
    return NextResponse.json({ error: "Missing signature" }, { status: 400 });
  }

  const stripe = getStripe();
  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(body, sig, secret);
  } catch (err) {
    console.error("[stripe/webhook] signature verification failed", err);
    return NextResponse.json({ error: "Invalid signature" }, { status: 400 });
  }

  try {
    switch (event.type) {
      case "checkout.session.completed": {
        const s = event.data.object;
        const userId = s.client_reference_id ?? s.metadata?.userId ?? null;
        const customerId = typeof s.customer === "string" ? s.customer : s.customer?.id;
        const subscriptionId = typeof s.subscription === "string" ? s.subscription : s.subscription?.id;
        if (userId && customerId) {
          await prisma.user.update({
            where: { id: userId },
            data: {
              stripeCustomerId: customerId,
              stripeSubscriptionId: subscriptionId ?? null,
              isPro: true
            }
          });
        }
        break;
      }
      case "customer.subscription.created":
      case "customer.subscription.updated": {
        const sub = event.data.object;
        const customerId = typeof sub.customer === "string" ? sub.customer : sub.customer.id;
        const active = sub.status === "active" || sub.status === "trialing";
        await prisma.user.updateMany({
          where: { stripeCustomerId: customerId },
          data: {
            isPro: active,
            stripeSubscriptionId: sub.id
          }
        });
        break;
      }
      case "customer.subscription.deleted": {
        const sub = event.data.object;
        const customerId = typeof sub.customer === "string" ? sub.customer : sub.customer.id;
        await prisma.user.updateMany({
          where: { stripeCustomerId: customerId },
          data: { isPro: false, stripeSubscriptionId: null }
        });
        break;
      }
      // `invoice.payment_failed` is optional — surface dunning state if desired.
      default:
        // Unhandled event type — return 200 so Stripe stops retrying.
        break;
    }
  } catch (err) {
    console.error("[stripe/webhook] handler failed", { type: event.type, err });
    return NextResponse.json({ error: "Handler failed" }, { status: 500 });
  }

  return NextResponse.json({ received: true });
}
```

**Important details:**

- `request.text()` — Stripe's signature is computed over the *raw* body; calling `.json()` first will mutate whitespace and break verification.
- `headers()` is awaited (Next 16 async dynamic API).
- `updateMany({ where: { stripeCustomerId } })` — single user per customer, but `updateMany` doesn't throw on zero matches (vs. `update` which would 404 a recovering webhook retry where the customer hasn't been linked yet).
- The `checkout.session.completed` case is the only path that links a brand-new `stripeCustomerId` to the `User` row (via `client_reference_id`). After that, the subscription events take over and use `stripeCustomerId` as the join key.
- Return 200 on unknown event types — otherwise Stripe will retry-spam.
- **Do not add rate limiting** to this route. Stripe's webhook IPs are documented and the signature itself authenticates the request.

### 5.5 `src/components/settings/billing-card.tsx` — Settings UI

Client component rendered inside the new Billing section on `/settings`. Renders two states:

- **Free user**: heading "Free plan", subtitle with current limits/usage, two `PendingButton`s — "Subscribe — $8/mo" (monthly) and "Subscribe — $72/yr" (yearly). Both submit forms posting to `createCheckoutSessionAction`.
- **Pro user**: heading "Pro plan — Monthly" / "Pro plan — Yearly" (period inferred client-side from session prefetch, see §6.2 — or pass `currentPeriod` via the page server component reading from Stripe), subtitle "Manage your subscription, payment method, and invoices in the Stripe portal.", single `PendingButton` — "Manage subscription" → posts to `createPortalSessionAction`.

Pattern is identical to [src/components/settings/change-password-dialog.tsx](../src/components/settings/change-password-dialog.tsx) — use `useTransition` + `PendingButton` + `toast.error()` on `{ error }` response.

Both forms are `<form action={...}>` with a hidden `<input>` carrying the period — the action `redirect()`s on success so no client-side navigation is needed.

```tsx
"use client";
import { useTransition } from "react";
import { toast } from "sonner";
import { PendingButton } from "@/components/ui/pending-button";
import { createCheckoutSessionAction, createPortalSessionAction } from "@/actions/billing";

export function BillingCard({ isPro }: { isPro: boolean }) {
  const [pending, startTransition] = useTransition();

  if (isPro) {
    return (
      <form action={() => startTransition(async () => {
        const result = await createPortalSessionAction();
        if (result?.error) toast.error(result.error);
      })}>
        {/* Pro plan headline + Manage subscription button */}
        <PendingButton pending={pending} type="submit">Manage subscription</PendingButton>
      </form>
    );
  }

  return (
    <div>
      {/* Free plan headline + two upgrade buttons (monthly / yearly) */}
      <form action={() => startTransition(async () => {
        const result = await createCheckoutSessionAction("monthly");
        if (result?.error) toast.error(result.error);
      })}>
        <PendingButton pending={pending} type="submit">Upgrade — $8/mo</PendingButton>
      </form>
      <form action={() => startTransition(async () => {
        const result = await createCheckoutSessionAction("yearly");
        if (result?.error) toast.error(result.error);
      })}>
        <PendingButton pending={pending} type="submit">Upgrade — $72/yr</PendingButton>
      </form>
    </div>
  );
}
```

```diff
+        const dbUser = await prisma.user.findUnique({
+          where: { id: token.id as string },

+        });
+        token.isPro = dbUser?.isPro ?? false;
+      }

     },
     session({ session, token }) {
       if (token.id) {
         session.user.id = token.id;
       }
+      session.user.isPro = token.isPro ?? false;
       return session;
     }
```

> **Why dynamic `await import("@/lib/prisma")` instead of a top-level import?** `auth.config.ts` is consumed by the edge middleware (via the split-config pattern from [src/auth.ts](../src/auth.ts)) and Prisma's Neon adapter doesn't run in the edge runtime. Dynamic import means the Prisma module is only resolved when the callback actually runs in the Node runtime (auth route handlers, server actions, server components) — never inside the middleware. If this turns out to break (because Next bundles the dynamic import eagerly), the alternative is to move the `jwt` callback into [src/auth.ts](../src/auth.ts) (the Node-only override) and leave the edge config minimal.

```diff
+import { getStripe } from "@/lib/stripe";

 export async function deleteAccountAction(): Promise<DeleteAccountResult | void> {

+    } catch (err) {

+    }
+  }

 }
```

| Export (JSON/ZIP) | ✗ (deferred) | ✓ (deferred) | — not yet built |
| Read existing data above limit | ✓ (downgrade keeps data visible/read-only) | ✓ | No read enforcement — only writes are gated |

**Downgrade behavior**: When a Pro user cancels and the subscription expires, `customer.subscription.deleted` flips `isPro: false`. All existing items/collections stay readable. The next *create* hits the Free-tier check and fails. Files/images previously uploaded stay in R2 and downloadable (don't break paid-for content retroactively).

**UI hints**: Wherever a Free user could trigger a paywall, surface the limit *before* the create attempt — disabled CTA + tooltip is gentler than letting them fill out a dialog and toast on submit. Specifically:

- `NewCollectionDialog` — same pattern, gated by collection count.

## 8. Webhook → JWT Session Sync

The session-stays-stale problem and its fix are covered in §6.2. To re-state for clarity:

**Problem.** With JWT strategy, the session token is signed by the server and lives in a cookie. When a Stripe webhook updates `isPro: true` in the DB, the user's existing cookie still says `isPro: false` — and `useSession()`'s `update()` only re-runs the `jwt` callback with `trigger === "update"`, which doesn't help here (the user isn't the one triggering the change).

**Fix.** Make the `jwt` callback **re-read `isPro` from the DB on every invocation**. The callback runs on every server-side session validation (every server action, every server component render, every `auth()` call). One extra `SELECT isPro` per call is cheap. After Stripe's webhook lands, the user's next page load picks up the new value.

**Cost.** Roughly one extra DB round-trip per server-rendered page. Mitigation if it ever becomes hot: cache the `isPro` lookup in Redis with a short TTL (e.g., 30s) keyed by `userId`. But ship without the cache first — measure before optimizing.

**Alternative considered: `unstable_update` server-side.** Stripe webhook could call NextAuth's `unstable_update` to flip the session — but that only works for **the current request's session**, not arbitrary user sessions, and the webhook has no session at all. Doesn't apply here.

**Verifying the sync in the UI.** The marketing CTA at `/register` redirects to `/sign-in`. After checkout, the user is redirected back to `/settings?checkout=success`. That page render goes through `auth()` → `jwt` callback → fresh DB read → updated session. The Billing card flips to the "Manage subscription" view on the same response. **No client-side polling, no `useSession.update()`, no reload needed.**

## 9. Implementation Order

Sequential — each step builds on the previous, and earlier steps can be merged independently behind the existing "all users have Pro" dev mode.

1. **Schema verify + env vars** (§3, §4). No code changes — just confirm `.env` has Stripe vars, Stripe Dashboard product/prices/webhook configured, Stripe CLI working locally. Fix the `.env.example` typos.
2. **Install Stripe SDK** (§6.11). `npm install stripe`.
3. **Create `src/lib/stripe.ts`** (§5.1). Singleton + price-ID accessors.
4. **Augment NextAuth session type + JWT callback** (§6.2, §6.3). Now every server action has `session.user.isPro`. **At this point existing pages don't change behavior** because nothing reads `isPro` yet — safe to ship behind a no-op merge.
5. **Create webhook handler** (§5.4). Test locally with `stripe trigger checkout.session.completed` and `stripe trigger customer.subscription.deleted`. Verify `isPro` flips in DB.

8. **Create `src/lib/billing.ts` + tests** (§5.2, §5.7).

11. **Cancel-on-delete** (§6.8). Closes the billing leak when users delete accounts.
